# 戦略MG研修 Web入力・自動計算システム 仕様書

> 対象: 戦略MG 製造業版入門編（ケーキ屋経営）
> バージョン: ドラフト v0.1（2026-06-24）
> ステータス: 仕様確定前のレビュー用。モック `mock/index.html` とセットで確認する。

---

## 1. 概要・目的

### 1.1 背景
戦略MG（マネジメントゲーム）研修では、ゲーム進行中の取引や決算を **紙の様式に手書き** し、電卓で集計している。
転記ミス・計算ミスが起きやすく、集計に時間がかかる。

### 1.2 目的
紙の3様式をWeb化し、**数値を入力すると関連する計算が自動で連鎖する** ようにする。
受講者は「入力」に集中でき、講師は集計・チェックの手間を減らせる。

### 1.3 対象様式（`docs/` のPDF）
| 様式 | ファイル | 役割 | Web化の扱い |
|---|---|---|---|
| ルール表 | `戦略MG製造業版入門編(ケーキ店経営）ルール表.pdf` | ゲームのルールA/B・生産能力・販売能力 | **参照ビュー**（入力なし） |
| 決算書 | `入門編MG決算書.pdf` | 現金出納帳・STRAC(P/L)・B/S・税・在庫/原価・各勘定 | **中核の入力＋自動集計** |
| 経営計画書 | `戦略MG製造業版入門編(ケーキ店経営）｢経営計画書」.pdf` | MQ会計による事前の経営計画 | **入力＋自動計算** |

### 1.4 参考サイト
`https://ec-mobile.netlify.app`（"EXECUTIVE COMPASS 自動決算"）。同種のMG決算自動化SPA。
本システムもこれに近い、レスポンシブで自動計算されるWebアプリを目指す。

### 1.5 決定済みの方針
- 対象様式: 決算書・経営計画書・ルール表の3つすべて
- デバイス: レスポンシブ（スマホ／PC両対応）
- データ保存: localStorage のみ（＋JSONエクスポート／インポート）。バックエンド・ログインなし
- 技術: React + Vite + TypeScript + Tailwind CSS、Netlify配信

## 2. 用語とMQ会計

戦略MGの会計はMQ会計（変動損益計算 / STRAC）に基づく。1個あたりと全体で対応する。

| 記号 | 意味 | 定義 |
|---|---|---|
| **P** | 売上単価 | ケーキ1個の販売価格 |
| **vP** | 変動単価（原価） | ケーキ1個あたりの変動費（材料原価） |
| **mP** | 粗利単価（付加価値単価） | `mP = P − vP` |
| **Q** | 販売数 | 売れた個数 |
| **PQ** | 売上高 | `PQ = P × Q` |
| **vPQ** | 変動費 | `vPQ = vP × Q`（＝売上原価） |
| **mPQ** | 粗利益（付加価値） | `mPQ = mP × Q = PQ − vPQ` |
| **F** | 固定費 | 人件費＋販売費＋管理費＋減価償却 |
| **G** | 経常利益 | `G = mPQ − F` |

派生指標:
- 原価率 = `vPQ / PQ`
- 粗利率（付加価値率）= `mPQ / PQ`
- 損益分岐点比率 = `F / mPQ`

## 3. 画面構成

### 3.1 ナビゲーション
- 上部に **タブ**: 「経営計画書」「決算書」「ルール表」
- ヘッダーに **会社（社名・社長名）** と **期（第◯期）** のセレクタ
- 会社・期は localStorage に保存。複数会社・複数期を切替できる

### 3.2 経営計画書 画面
ゲーム前の計画。入力すると右側・下部のSTRAC図が自動更新される。

- 入力: ①G利益目標、②F固定費の各内訳、④P単価・vP変動単価、当期
- 自動: ②F合計、③mPQ粗利、⑤mP粗利単価、⑥Q必要販売数、⑦PQ売上高、⑧vPQ変動費、原価率/粗利率/損益分岐点比率
- 表示: 「1個あたりコスト構造」図、「変動損益計算書」箱ダイアグラム（PQ＝mPQ＋vPQ、mPQ＝F＋G の面積表現）

### 3.3 決算書 画面
ゲーム後の集計。中核。以下のブロックで構成。

1. **基本情報**: 第◯期・社名・社長名
2. **借入金可能枠**: 純資産×倍率＝枠、現在借入残高、今期借入可能額、金利・期首支払金利
3. **現金出納帳**: 行ごとに取引を入力。列は下表。現金残高は累積自動計算
4. **STRAC（変動損益計算書・P/L）**: 出納帳から自動集計
5. **法人税・利益剰余金の計算**
6. **貸借対照表（B/S）**: 差額=0チェック
7. **補助勘定**: 現金勘定・什器・借入金・在庫(棚卸)/原価計算

### 3.9 組織コード・組織タブ（確定 2026-06-30）
- 会社情報に**組織コード**を入力。ナビに**「組織」タブ**を追加。

### 3.8 決算書の追加（確定 2026-06-30）
- **B/S 図解**: 貸借対照表を、資産（現金・在庫・什器）｜負債・純資産（未払法人税・借入金・資本金・利益剰余金）の**同じ高さの積み上げ棒**で対比し、貸借一致を可視化。
- **キャッシュフロー計算書**: 営業CF（売上＋保険金−材料−人件費−販売費−管理費−納税）／投資CF（−什器）／財務CF（資本金＋借入−返済）に区分。期首現金＋現金増減＝期末現金（必ず B/S の現金に一致）。
- **振り返り**: KPI推移（経常利益G・売上PQ・粗利率・損益分岐点比率）を**折れ線グラフ**で表示（G は0基準線つき）。

## 4. データモデル

```
Company（会社）
  id, 社名, 社長名
  periods: Period[]（期）

Period（期）
  期番号（1..5）
  plan: PlanInput          // 経営計画書の入力
  cashRows: CashRow[]      // 現金出納帳の各行
  adjustments: PeriodAdjustments // 期末調整（什器台数, 会社盤在庫数, 特別損益, 法人税上書き 等）
  opening: OpeningBalances // 前期繰越（現金, 在庫個数&金額, 什器, 借入金, 資本金, 利益剰余金）

CashRow（出納帳1行）
  勘定科目ラベル
  入金: { 資本金, 借入金, 売上, 受取保険金 }
  出金: { 什器, 材料仕入, 人件費, 販売費, 管理費, 借入金返済, 納税 }
  売上個数, 仕入個数
```

- **前期繰越の連鎖**: 期Nの「次期繰越」（現金・在庫・什器・借入金・資本金・利益剰余金）を期N+1の `opening` に自動転記。
  - **在庫は材料・製品を分けて引き継ぐ**（確定 2026-06-30）：期末の盤面で材料は材料（rawCubes）、製品は製品（products）として次期へ。金額（在庫評価額＝平均原価×個数）は合算のまま転記しB/Sは不変。
  - **商品開発チップ・広告チップは2枚以上あれば1枚だけ次期へ繰越**（確定 2026-06-30）。教育・保険チップは期末返却で0。
- **永続化**: localStorage キー `mg-system:v1` に会社配列をJSONで保存。エクスポート＝同JSONのダウンロード、インポート＝読み込みで上書き/追加。

## 5. 計算仕様

> 純粋関数として `src/lib/calc/` に実装し、Vitestでテスト。UIから独立。

### 5.1 経営計画書（`plan.ts`）

```
F固定費 =
    人件費(スタッフ人数 × 給料[当期])
  + 採用(5 × 新規採用人数)
  + 広告(10 × 枚)
  + 商品開発(20 × 枚)
  + 家賃(25)
  + 教育(20 × 枚)
  + 保険(5 × 枚)
  + 金利(借入残高 × 0.05)
  + 減価償却(10 × 什器台数)

mPQ粗利 = G + F
mP粗利単価 = P − vP
Q必要販売数 = ⌈ mPQ ÷ mP ⌉      // 小数切り上げ
PQ売上高 = P × Q
vPQ変動費 = vP × Q
原価率 = vPQ / PQ
粗利率 = mPQ / PQ
損益分岐点比率 = F / mPQ
```

### 5.2 決算書（`settlement.ts`）

**列合計**: 入金 ア/イ/ウ/A、出金 エ/オ/カ/キ/ク/ケ/コ を縦集計。
**現金残高**: 各行 = 前行残高 ＋（その行の入金合計）−（その行の出金合計）。先頭は前期繰越現金。

**在庫・原価計算（⑫）**（金額はすべて整数・**平均単価を四捨五入**してから按分）:
```
合計個数 = 前期繰越個数 + 仕入個数(オ)
合計金額(③) = 前期繰越在庫金額 + 材料仕入金額(オ)
平均単価(④) = round(合計金額 ÷ 合計個数)        ※四捨五入の整数
廃棄等(⑥)   = 平均単価 × 廃棄個数
次期繰越(⑤) = 平均単価 × 期末在庫個数
売上原価(⑦=vPQ) = 合計金額 − 次期繰越 − 廃棄等   ※残余（③−⑤−⑥）。⑤+⑥+⑦=③を厳密に保ちB/S均衡
次期繰越在庫(個) = 合計個数 − Q − 廃棄個数
廃棄等(個) = 合計個数 − Q − 会社盤在庫数（差異がある場合）
```
> **端数処理（2026-06-30 確定）**: 小数が出るのは平均単価のみ。平均単価を四捨五入し、廃棄・次期繰越を「平均単価×個数」、売上原価を残余（③−⑤−⑥）とすることで、以降の vPQ/mPQ/G/特別損益/当期純利益・B/S をすべて整数化する。表示も四捨五入（小数を出さない）。

**STRAC（P/L）**:
```
PQ = 売上合計(ウ)
Q  = 売上個数合計
P  = PQ / Q
vPQ = 売上原価（⑫より）
vP  = vPQ / Q
mPQ = PQ − vPQ
mP  = P − vP
F   = 人件費(カ) + 販売費(キ) + 管理費(ク) + 減価償却(什器台数 × 10)
損益分岐点比率 = F / mPQ
G(経常利益) = mPQ − F
```

**補助勘定**:
```
現金勘定:  次期繰越 = 前期繰越 + 入金合計 − 出金合計
什器:      次期繰越 = 前期繰越(什器) + 什器購入(エ) − 減価償却(什器台数×10)
借入金:    次期繰越 = 前期繰越(借入) + 借入(イ) − 返済(ケ)
```

**法人税・利益剰余金**:
```
税引前当期利益 = G + 特別損益
法人税等 = 税引前当期利益 × 30%   // 最低納税額5、PDF注記の符号条件に従う。手動上書き可
当期利益 = 税引前当期利益 − 法人税等
次期繰越利益剰余金 = 前期繰越利益剰余金 + 当期利益
```

**貸借対照表（B/S）**:
```
資産合計 = 現金(現金勘定の次期繰越) + 在庫 + 什器
負債合計 = 未払法人税等 + 借入金
純資産合計 = 資本金 + 利益剰余金
チェック: 資産合計 − (負債合計 + 純資産合計) = 0 ならOK
```

> 注: 法人税の符号条件・棚卸差異の細部はPDF（`docs/入門編MG決算書.pdf`）の注記に分岐がある。
> 実装時に注記どおりに実装し、PDFの数値例をユニットテストで固定する。

|---|---|
| 広告 | 10 / 枚（販売能力+2、有効は販売スタッフ1人につき2枚まで／購入枚数は無制限、期末1枚返却） |
| 商品開発（メニュー開発） | 20 / 枚（1回1枚、価格競争力2、期末1枚返却） |
| 教育 | 20 / 枚（製造能力+1、期末返却） |
| 保険チップ | 5 / 枚（事故対応、期末返却） |
| スタッフ採用 | 採用費 5 / 人（製造能力2・販売能力2） |
| 什器購入 | 100 / 台（減価償却 10 / 台、製造スタッフ2人まで稼働） |
| 配置転換 | 5（製造⇄販売スタッフの移動） |
| ストッカー材料単価 | 16 |
| 給料テーブル（1人あたり, 期別） | 1期25 / 2期28 / 3期31 / 4期34 / 5期37 |
| 家賃 | 25 / 期 |
| 法人税 | 税率30%、最低納税額5 |
| 在庫上限 | 15 個 / 店舗陳列上限 15 個 |

**生産能力**: 製造には**機械（什器）と製造スタッフの両方が必要**。機械1台で製造スタッフ2人が稼働（機械orスタッフが0なら製造不可。例：機械1×スタッフ3なら2人だけ稼働し1人は遊休）。稼働1人あたり2個、教育で3個。
**販売能力**: 販売スタッフ1人＋広告2枚まで有効＝最大6販売。

## 7. 画面制御・バリデーション

記帳・各入力フォームの制御方針。基本は**モーダル（入力フォーム）内でリアルタイム判定**し、NG時は理由を赤字表示して「決定して記帳」ボタンを無効化する。

### 7.1 方針（確定 2026-06-30）
- **資金（現金残高）**: 制御しない。手元資金を超える支出・マイナス残高も記帳可。ただし**残高がマイナスの行・合計・ヘッダの現金残高は赤色で警告表示**（情報表示であり記帳は止めない）。
- **能力・上限の超過**: ブロック（記帳させない）。
- **NG時の表示**: モーダル内に理由を赤字表示＋「決定して記帳」ボタンを無効化。入力変更でリアルタイム再判定。
- 既存の「製造/販売/配置転換を黙ってクランプ」は廃止し、上限超過は**決定無効化＋理由表示**に統一する。

### 7.2 入力フォーマット
- 数量系（個数・台数・人数・枚数）= **1以上の整数**（0・負・小数は不可）。
- **固定フィールド（編集不可）**: 商品開発の枚数＝**1固定**、教育の枚数＝**1固定**（入力欄は無効化表示。実値は常に1）。
- 金額系（開業資本金・増資・借入・返済・管理費・販売費・売価P）= **0以上の整数**（負・小数は不可）。
- 仕入れの個数・借入返済の金額は**初期値0**（ユーザーが入力）。仕入れ単価 = 10〜16 のドロップダウン（初期10）。期 = 到達済みの期から選択。
- 配置転換は**人数**で指定（旧「回数」を改称、コスト5/人）。
- **複数行記帳（仕入れ・販売、確定 2026-06-30）**: 1モーダル＝1記帳のまま、モーダル内の「＋行を追加」で**複数の（数量×単価）行**を入力できる。出納帳・PDFには**1記帳（合計額＋内訳）**として1行で記帳。バリデーションは各行の数量＝1以上整数／単価（仕入＝10〜16のプルダウン・売価P＝0以上整数）、上限は**合計数量**で判定（仕入＝在庫＋合計≤15、販売＝合計≤販売能力かつ製品在庫以内）。
- 借入倍率 = 0以上の整数（講師設定）。返済率 = 0〜100 の整数（講師設定）。

### 7.3 在庫・陳列の保有上限（ブロック・累計）
- **仕入れ**: 材料在庫 `rawCubes + 個数 ≤ 15`。
- **製造**: 店舗陳列 `products + 個数 ≤ 15`（7.4・材料在庫の範囲内も同時に満たすこと）。

### 7.4 能力の上限（ブロック・「1回の操作数」に対して）
能力式は会社盤の表示と一致させる。
- `workers = min(製造スタッフ, 機械×2)`（製造には機械必須・機械1台で2人稼働。機械orスタッフが0なら workers=0＝製造不可）
- `製造能力 mfgCap = workers × (教育>0 ? 3 : 2)`
- `販売能力 salesCap = 販売スタッフ×2 + min(広告, 販売スタッフ×2)×2`

ルール:
- **製造（1回）**: **機械・製造スタッフが0なら製造不可**。個数 ≤ 製造能力 `mfgCap`、かつ 個数 ≤ 手持ち材料 `rawCubes`、かつ 7.3 の陳列上限。
- **販売（1回）**: 個数 ≤ 販売能力 `salesCap`、かつ 個数 ≤ 手持ち製品 `products`。
- **広告**: 購入枚数の制限なし（販売スタッフ数に関係なく購入可）。ただし**販売能力への寄与は販売スタッフ×2枚まで**有効（`salesCap` の `min(広告, 販売スタッフ×2)` で頭打ち）。
- **配置転換**: 移動人数 ≤ 該当スタッフ数。
- **借入返済**: 返済額 ≤ 借入残高（残高を超える返済は不可。無借金時は0のみ）。
- **教育**: 保有上限1（`現在の教育チップ + 枚数(=1) ≤ 1`、累計判定）。
- **スタッフ退職（イベント）**: 退職する区分（製造／販売）のスタッフが0人なら記帳不可。

### 7.5 状態・順序（既存踏襲）
- 決算後（settled）: 記帳の追加・編集・削除をロック。
- 期末処理計上後（closingPrep）: 記帳追加をブロック（「記帳に戻る」で解除）。
- 第1期: 借入・借入可能枠・金利・期末返済はなし（第2期以降のみ）。
- 借入: 借入可能枠（純資産×倍率 − 現在残高）以内、第1期は不可。
- 自動計上行（給料/家賃/期首納税/借入金利/期末返済）: 金額のみ編集可・個別削除不可（派生行＝借入金利/期末返済は編集も不可）。

### 7.6 NG時の表示（画面制御）
- モーダル下部にNG理由を赤字で表示し、「決定して記帳」ボタンを `disabled` にする。
- 入力（数量・金額・選択）変更のたびに再判定し、OKになれば即時に有効化。
- 1フォームに複数のNGがある場合は、該当理由をまとめて表示する。

## 8. 本番アーキテクチャ（確定 2026-07-01）

### 8.1 方針
- **講師（管理者）のみログイン**。参加者は**ログイン不要**で、講師が発行する**組織コード付きURL**から参加する（Kahoot型）。
- 講師の状況確認は**「更新」ボタンで取得**（常時リアルタイム配信は不要）。→ WebSocket等は必須ではなく、**REST（GET/POST）で実装可**。

### 8.2 技術構成
- フロント: **React + Vite + TypeScript + Tailwind CSS**（Netlify等に配信）。
- **計算ロジックはモックの純粋関数をそのままTS化して流用**（`recompute`／`settle`／STRAC・B/S・CF・棚卸）。クライアントと管理者で共有し、Vitestで数値例（給料表・固定費・平均原価四捨五入・B/S差0・棚卸差異0）を固定。
- バックエンド: **Supabase（DB＋講師Authのみ）** が最短。または軽量な自前API（Hono/Express＋Postgres）。リアルタイム機能は使わない。

### 8.3 認証・参加フロー
1. 講師がログイン → 管理画面で**組織コードを決め参加用URL（`/?org=コード`）を発行・配布**。
2. 参加者はURLを開く（ログイン不要）→ 組織コードは自動設定 → **会社名・社長名を入力して開始**。
3. 参加者の操作はバックエンドに保存（毎手番／決算時）。会社は「組織コード＋会社名」で識別。
4. 講師は管理画面で「更新」→ 参加者一覧・各社の状況・組織比較を確認（参加者の画面を閲覧専用で表示）。
- セキュリティ: 組織コード（＋必要ならトークン）は共有キー（対面研修向け）。**講師ログインで管理・集計・URL発行を保護**、参加者の書き込みは自組織スコープに限定（Supabase RLS／トークン検証）。

### 8.4 データモデル（モックの localStorage → DB）
- 研修（組織コード）→ 会社（参加者）→ 期（period）→ 記帳（entries）・各期確定値（period_results）。
- モックの `mgLive[org][company]`（現在状態）→ `companies`＋`entries`、`mgOrg[org][company].history`（各期成績）→ `period_results`。

### 8.5 モックでの再現（`mock/`、バックエンド無し）
- 参加者参加URL `index.html?org=コード` → 組織コードを自動設定（読み取り専用）。
- 管理者 `admin.html`: **講師ログイン（デモPW）** → 組織選択／**参加用URL発行**／参加者一覧／参加者の画面を `index.html?vorg=&vco=` の iframe で閲覧専用表示／「更新」で取得。
- 共有は**同一ブラウザの localStorage** のみ（本番ではサーバ＋講師認証に置き換え）。
- **リロード自動復元（確定 2026-07-01）**: 参加者が記帳中にリロード／再訪しても、`saveLive()` が毎操作で保存する `mgLive` と自己識別 `mgMe={org,name}` から**自動で続きを復元**（`switchTab(settled?'statement':'play')`）。復元通知バナー（「新しく始める」で破棄＝当該社の `mgLive` 削除＋リロード）付き。**限界**: 同一ブラウザの localStorage のみ。ブラウザのデータ削除・別端末・シークレットでは復旧不可 → **本番はサーバ保存（Supabase等）で「組織コード＋会社名」により端末跨ぎ・削除耐性のある完全復旧**とする。

### 8.6 未決仕様の確定（2026-07-01・すべて「シンプル維持」で決定）
| 論点 | 決定 | 補足 |
|---|---|---|
| 市場・価格の仕組み | **各社の自由入力のまま**（入門編・市場/入札は再現しない） | 各社が独立して価格を決めて売買。計算エンジン変更なし |
| イベントカードの出し方 | **手動選択のまま**（参加者/講師がカードを選ぶ） | 卓上の物理カードに合わせて手入力。ランダム抽選/講師配布は実装しない |
| 参加者の識別（本番・ログインなし） | **会社名のみ**（URL＋会社名入力） | 「組織コード＋会社名」で識別。なりすましは基本許容（対面研修前提）。PIN・会社枠発行は不要 |
| 期の進行（次の期へ） | **各社が自由に進む**（現状維持） | 講師の一斉進行・期ロックは実装しない |
| 手番・行動回数の上限 | **縛らない**（自由進行・ターン数/意思決定はカウントのみ） | |
| 保存タイミング（本番） | **期末決算の確定時に保存**（講師は「更新」で最新取得） | |

→ 上記より、講師機能は「一斉進行」「PIN認証」「市場・イベントエンジン」を作らず、**成績の集計・エクスポート・組織/参加者の管理（クリア/リセット）**に絞る。
